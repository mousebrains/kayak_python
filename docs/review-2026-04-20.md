## How to cross-check

Each finding has a **Reproduce** block. Commands are one of three kinds:

- `local$` — run from this repo on any workstation (inspects checked-in code).
- `test$` — run on `levels-test.wkcc.org` (prod config + live DB).
- `web$` — HTTP request against `https://levels-test.wkcc.org/`.

A finding is "confirmed" when the Reproduce output matches the **Expect** line.
If the test box has drifted from `main`, some fixes may already be in place —
that's fine, note the drift in the checkbox comment.

## TIER 1 — Reliability / data integrity (do first)

### T1-1. Pipeline swallows all stage failures → systemd reports success on data loss

- **File:** `src/kayak/cli/pipeline.py:70-77`
- **Evidence:** broad `except Exception` per step, logger.error, then continues. No exit code propagation.
- **Why:** a dead upstream (USGS, NWRFC) produces observation gaps with no alert. OnFailure email never fires.
- **Reproduce:**
  ```
  local$ sed -n '65,90p' src/kayak/cli/pipeline.py
  test$  sudo systemctl start kayak-pipeline.service  # after introducing a failing step
  test$  systemctl show kayak-pipeline.service -p ExecMainStatus  # expect: 0 (wrong)
  ```
- **Expect:** `ExecMainStatus=0` even when a step fails.
- **Fix:** collect step failures into a list; after the loop, `raise SystemExit(1)` if any failed. Add `--continue-on-error` flag for manual recovery runs. Add 2 tests: exit code on failure, flag behavior.
- **Effort:** 2 h.

### T1-2. PHP `busy_timeout=5000` vs Python `busy_timeout=30000` → user-visible 503s during hourly writes

- **Files:** `php/includes/db.php:21` (PHP = 5 s), `src/kayak/db/engine.py:17` (Python = 30 s)
- **Why:** during the hourly pipeline the Python side holds the write lock up to 30 s; PHP times out at 5 s and returns HTTP 503 "Service temporarily unavailable".
- **Reproduce:**
  ```
  local$ grep -n "busy_timeout" php/includes/db.php src/kayak/db/engine.py
  test$  ( while true; do curl -s -o /dev/null -w '%{http_code}\n' https://levels-test.wkcc.org/api.php?id=1; sleep 1; done ) &
  test$  sudo systemctl start kayak-pipeline.service   # then watch the curl loop for 503s
  ```
- **Expect:** PHP is 5000, Python is 30000; occasional 503 during pipeline runs.
- **Fix:** change PHP to `PRAGMA busy_timeout=30000`.
- **Effort:** 15 min.

### T1-3. No schema migration system — only `Base.metadata.create_all()`

- **File:** `src/kayak/cli/init_db.py`
- **Why:** create_all is idempotent for new tables but cannot alter columns, change constraints, or add indexes on existing tables. Production schema diverges from models.py with no tracked record of what's applied.
- **Reproduce:**
  ```
  local$ ls data/db/migrations/ 2>/dev/null; ls src/kayak/cli/migrate.py 2>/dev/null
  test$  sqlite3 $DB ".schema schema_migrations" 2>&1 | head -3
  test$  sqlite3 $DB ".schema observation" | grep -i check
  local$ sqlite3 :memory: ".schema observation" # diff against prod
  ```
- **Expect:** no migrations dir, no `schema_migrations` table, no CHECK constraints.
- **Fix:** adopt Alembic, or a homegrown `levels migrate` + `data/db/migrations/NNNN_*.sql` tracked in a `schema_migrations(version TEXT PK, applied_at DATETIME)` table. Wire into pipeline.
- **Effort:** ½ day. Unblocks T1-4, T1-6, and schema changes going forward.

### T1-4. Missing CHECK / XOR constraints on Source, ReachLevel, ReachClass

- **File:** `src/kayak/db/models.py:170-174,498-518,533-546`
- **Why:** `Source.fetch_url_id` and `calc_expression_id` are both nullable with no XOR constraint — a row can have both or neither, violating design intent. `ReachLevel.low/high` and `ReachClass.low/high` have no `low <= high` check.
- **Reproduce:**
  ```
  local$ grep -n "CheckConstraint" src/kayak/db/models.py
  test$  sqlite3 $DB "SELECT COUNT(*) FROM source WHERE fetch_url_id IS NULL AND calc_expression_id IS NULL"
  test$  sqlite3 $DB "SELECT COUNT(*) FROM source WHERE fetch_url_id IS NOT NULL AND calc_expression_id IS NOT NULL"
  test$  sqlite3 $DB "SELECT COUNT(*) FROM reach_level WHERE low > high"
  test$  sqlite3 $DB "SELECT COUNT(*) FROM reach_class WHERE low > high"
  ```
- **Expect:** zero CheckConstraint in models.py; the COUNT queries may already be 0 (data is fine) but the constraint isn't enforced.
- **Fix:** add in first migration after T1-3:
  - `ALTER TABLE source ADD CHECK ((fetch_url_id IS NULL) != (calc_expression_id IS NULL));`
  - `ALTER TABLE reach_level ADD CHECK (low <= high);`
  - `ALTER TABLE reach_class ADD CHECK (low <= high);`
  - mirror with `CheckConstraint(...)` in `__table_args__`.
- **Effort:** 1 h (after T1-3).

### T1-6. Caches can diverge — decimate/merge vs `latest_observation` / `latest_gauge_observation`

- **Files:** `src/kayak/cli/decimate.py`, `src/kayak/db/data_db.py` (merge_sources near line 648-654; update_latest at 282-293, 412-423)
- **Why:** decimation deletes observation rows without touching `latest_observation`; merge DELETEs in the target's time range, then re-INSERTs — if it crashes between, the target has neither.
- **Reproduce (detect divergence on live DB):**
  ```
  test$  sqlite3 $DB <<'SQL'
    SELECT lo.source_id, lo.observed_at AS lo_time, o.observed_at AS obs_time
    FROM latest_observation lo
    LEFT JOIN observation o
      ON o.source_id = lo.source_id
     AND o.data_type = lo.data_type
     AND o.observed_at = lo.observed_at
    WHERE o.observed_at IS NULL
    LIMIT 20;
  SQL
  ```
- **Expect:** any rows returned = divergence present (cache points at a deleted observation).
- **Fix:** (a) wrap decimate in a transaction that rebuilds affected `latest_*` rows; (b) switch `merge_sources` to a staging-table swap, or use `INSERT ... ON CONFLICT` + delete of non-surviving rows.
- **Effort:** 1 day.

## TIER 2 — Security posture

### T2-9. No rate limit on magic-link issuance

- **File:** `php/login.php:46-67`
- **Why:** an attacker can flood any email address with login links.
- **Reproduce:**
  ```
  local$ grep -nE "rate|limit|throttle" php/login.php php/includes/auth.php
  web$   for i in $(seq 1 10); do curl -s -o /dev/null -w '%{http_code}\n' -d "email=attacker@example.com" https://levels-test.wkcc.org/login.php; done
  ```
- **Expect:** no throttle code; all 10 requests return the same status (likely 200).
- **Fix:** per-IP + per-email cap (e.g. 5/hour) stored in a small table keyed by hashed email + IP.
- **Effort:** 1 h.

### T2-10. CSRF token not regenerated after login

- **File:** `php/includes/auth.php:171-178`
- **Why:** the pre-auth cookie carries into the post-auth session, enabling fixation-style attacks.
- **Reproduce:**
  ```
  local$ grep -n "set_editor_session\|csrf_token" php/includes/auth.php
  web$   curl -si -c /tmp/cj1 https://levels-test.wkcc.org/login.php | grep -i 'editor_csrf'
  web$   # complete login with magic link in a second terminal, then:
  web$   curl -si -b /tmp/cj1 https://levels-test.wkcc.org/account.php | grep -i 'editor_csrf'
  ```
- **Expect:** same CSRF cookie value before and after login.
- **Fix:** regenerate CSRF inside `set_editor_session()` after successful auth.
- **Effort:** 30 min.

### T2-11. Open-redirect TOCTOU in login `next`

- **File:** `php/login.php:20,28-29`
- **Evidence:** `filter_input(INPUT_GET, 'next', FILTER_SANITIZE_SPECIAL_CHARS)` is used in one branch while `safe_next_url($_GET['next'])` is called elsewhere on the unfiltered original.
- **Reproduce:**
  ```
  local$ sed -n '15,35p' php/login.php
  web$   curl -si 'https://levels-test.wkcc.org/login.php?next=https://evil.example/'   | grep -i location
  ```
- **Expect:** code shows both paths; `Location:` header should *not* redirect off-site (if it does, finding is severe).
- **Fix:** drop `filter_input`; use `safe_next_url()` exclusively. Add a test with `next=https://evil.example/`.
- **Effort:** 15 min.

### T2-12. edit.php HTTP Basic Auth brute-force guard depends on an nginx `limit_req` zone that may not be sourced on clean installs

### T2-13. Security response headers — verify on dynamic PHP output

- **Files:** `conf/*` nginx, `php/includes/header.php`
- **Reproduce:**
  ```
  web$   curl -sI https://levels-test.wkcc.org/           | grep -iE 'strict-transport|content-security|x-content-type|referrer-policy|permissions-policy|x-frame'
  web$   curl -sI https://levels-test.wkcc.org/login.php  | grep -iE 'strict-transport|content-security|x-content-type|referrer-policy|permissions-policy|x-frame'
  ```
- **Expect:** static and PHP responses should both have HSTS, CSP, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy`, `X-Frame-Options`. Mismatch = PHP-FPM path is bypassing the nginx `add_header` (not marked `always`).
- **Fix:** add `always` flag on `add_header` in nginx; or set headers in `header.php`.
- **Effort:** 1 h.
