# Cleaning up foreign-key violations on the live database

The live `kayak.db` has accumulated orphan rows that would fail `PRAGMA
foreign_key_check`. They're harmless at rest (SQLite doesn't enforce FKs
unless asked) but they confuse tools that do enforce them — including
`scripts/import_metadata.py` and anything that turns on `foreign_keys=ON`
before deleting a parent row.

This document walks through a one-time cleanup on the live server.

## Current state

Snapshot taken 2026-04-15 (via `scripts/db_pull.sh`):

| Table | Orphan rows | Parent |
|---|---:|---|
| `observation` | 209,602 | `source` |
| `reach_level` | 237 | `reach` |
| `reach_guidebook` | 106 | `reach` |
| `reach_state` | 86 | `reach` |
| `reach_class` | 81 | `reach` |
| `latest_observation` | 62 | `source` |
| `latest_gauge_observation` | 61 | `gauge` |
| `gauge_source` | 35 + 34 | `source` / `gauge` |
| `reach` | 1 | `gauge` (gauge_id should be NULL) |

Re-run the diagnostic any time with:

```bash
sqlite3 /home/pat/DB/kayak.db "PRAGMA foreign_key_check;" \
    | awk -F'|' '{print $1}' | sort | uniq -c | sort -rn
```

## Why it's safe to delete orphans

Every junction/cache table listed above has `ON DELETE CASCADE` on its
foreign keys, which means whenever a parent row is deleted, all child rows
should disappear automatically. The orphans only exist because at some
point parents were deleted with `foreign_keys=OFF` (so cascades didn't
fire). The orphans carry no useful information — their parent is gone
and there's nothing to reattach them to.

The single orphan `reach.gauge_id` is different: its FK is `ON DELETE SET
NULL`, meaning the gauge's deletion should have nulled the column but
didn't. Fix by setting it to `NULL` rather than deleting the reach.

## Cleanup procedure

Run this on the live server, as user `pat`. Budget a minute or two of
pipeline downtime while the cleanup runs.

### 1. Stop the pipeline and take a backup

```bash
for u in kayak-pipeline.timer kayak-decimate.timer kayak-backup.timer \
         kayak-pipeline.service kayak-decimate.service kayak-backup.service; do
    sudo -n systemctl stop "$u" 2>/dev/null || true
done

TS=$(date -u +%Y%m%dT%H%M%SZ)
mkdir -p ~/kayak/backups
sqlite3 ~/DB/kayak.db ".backup ~/kayak/backups/kayak-before-fkcleanup-${TS}.db"
gzip -9 ~/kayak/backups/kayak-before-fkcleanup-${TS}.db
ls -lh ~/kayak/backups/kayak-before-fkcleanup-${TS}.db.gz
```

If anything goes wrong, restore with:

```bash
gunzip -c ~/kayak/backups/kayak-before-fkcleanup-${TS}.db.gz > ~/DB/kayak.db
rm -f ~/DB/kayak.db-wal ~/DB/kayak.db-shm
```

### 2. Run the cleanup transaction

Save the following as `/tmp/fk_cleanup.sql` and run it with
`sqlite3 ~/DB/kayak.db < /tmp/fk_cleanup.sql`:

```sql
-- fk_cleanup.sql
PRAGMA foreign_keys = OFF;
BEGIN;

-- Orphan observations (parent source missing)
DELETE FROM observation
 WHERE source_id NOT IN (SELECT id FROM source);

-- Orphan cache rows (regenerated by next pipeline tick anyway)
DELETE FROM latest_observation
 WHERE source_id NOT IN (SELECT id FROM source);
DELETE FROM latest_gauge_observation
 WHERE gauge_id NOT IN (SELECT id FROM gauge);

-- Orphan gauge_source junction rows
DELETE FROM gauge_source
 WHERE source_id NOT IN (SELECT id FROM source)
    OR gauge_id  NOT IN (SELECT id FROM gauge);

-- Orphan reach junctions
DELETE FROM reach_state
 WHERE reach_id NOT IN (SELECT id FROM reach);
DELETE FROM reach_class
 WHERE reach_id NOT IN (SELECT id FROM reach);
DELETE FROM reach_level
 WHERE reach_id NOT IN (SELECT id FROM reach);
DELETE FROM reach_guidebook
 WHERE reach_id     NOT IN (SELECT id FROM reach)
    OR guidebook_id NOT IN (SELECT id FROM guidebook);

-- Reach rows with a now-missing gauge: FK says ON DELETE SET NULL,
-- so null them rather than delete the reach.
UPDATE reach
   SET gauge_id = NULL
 WHERE gauge_id IS NOT NULL
   AND gauge_id NOT IN (SELECT id FROM gauge);

-- Verify inside the transaction — if anything's still violating, roll back.
-- (foreign_key_check returns no rows when clean.)
SELECT 'STILL VIOLATING: ' || "table" || ' rowid=' || rowid
  FROM pragma_foreign_key_check;

COMMIT;

-- Reclaim space (observation alone dropped ~210k rows).
VACUUM;

PRAGMA foreign_keys = ON;
PRAGMA foreign_key_check;
PRAGMA integrity_check;
```

Expected output at the end:

```
ok
```

If the `STILL VIOLATING` SELECT inside the transaction prints anything,
investigate before running `COMMIT`. To abort mid-flight, send `Ctrl-C`
before the `COMMIT;` line executes, or run `ROLLBACK;` in an interactive
`sqlite3` session instead of piping the file.

### 3. Restart the pipeline

```bash
for u in kayak-pipeline.timer kayak-decimate.timer kayak-backup.timer; do
    sudo -n systemctl start "$u"
done
systemctl list-timers 'kayak-*'
```

### 4. Confirm and refresh

```bash
# Should output: ok  (repeated from the SQL but worth re-running)
sqlite3 ~/DB/kayak.db 'PRAGMA foreign_key_check;'

# DB size after VACUUM — expect ~100 MB smaller thanks to the deleted obs.
du -h ~/DB/kayak.db
```

The next pipeline tick will recompute `latest_observation` and
`latest_gauge_observation` from scratch; no explicit action needed.

## After cleanup

With `foreign_key_check` clean, you can:

- Run `scripts/import_metadata.py` without FK-violation warnings.
- Turn on `PRAGMA foreign_keys = ON` at connection time in the Python
  code if desired (currently it's off by default in `kayak.db.engine` —
  see `src/kayak/db/engine.py`). That would catch any *new* violations
  at write time rather than letting them accumulate silently.
- Re-run `scripts/db_pull.sh` + `scripts/export_metadata.py` locally to
  check in a clean metadata snapshot under `data/db/`.

## Why orphans accumulated

Likely causes:

- Legacy MySQL sync (`scripts/import_from_dump.py`, now retired) built
  the DB with `foreign_keys` disabled to speed up bulk inserts.
- Manual edits via `sqlite3` CLI don't enforce FKs unless the session
  ran `PRAGMA foreign_keys = ON` (off by default per connection).
- Row-level writes from the Python code don't enforce FKs either,
  because `kayak.db.engine` never turns them on.

Flip `PRAGMA foreign_keys = ON` in the engine setup once the DB is clean
to prevent regressions.
