### 0. Security & correctness sharp edges

- **HIGH [ops, latent] — `scripts/db_push.sh:134` runs `DELETE FROM pages;` against a table dropped 58 migrations ago.** `data/db/migrations/0006_drop_pages.sql` removed `pages` (no ORM model exists). In the remote merge heredoc (under `set -euo pipefail`), `sqlite3` errors `no such table: pages` and exits non-zero — **aborting the restore before the DB swap (`:159`) and the timer-restart loop (`:176–179`)**, so `kayak-pipeline`/`decimate`/`backup-*` timers (stopped at `:100–106`) never restart and the new DB is never installed. Any `db_push.sh` run strands prod on stale data + halted backups. Reproduced. Low real-world likelihood (the dev-only push script, told never to run), high impact. **Fix:** delete line 134 *and*, durably, wrap the stop→restart critical section (`:105`–`:178`) in a `trap '…systemctl start…' EXIT` so *any* failure there restarts the timers — line-deletion removes today's trigger but not the structural flaw.
- **MED [docs/trust] — `CHANGELOG.md` `[Unreleased]` carries false facts.** *(The PR-number omission is the file's stated curation policy — "curated and thematic … see git log for the exhaustive history" — not a defect; the real issue is wrong facts.)* `:114` states R1.5 "still runs a pat-writable binary as root … tracked as R1.5" — **#48 closed it** — and the same bullet brags it "rejected `javascript:`/`data:` URIs," which the tab-bypass (§0) shows is incomplete; `:95` says "634 entries" (baseline is 104/218); `:102` says "floor 5% → 55%" (now **58%**, #52). A closed security HIGH described as residual is the trust hit. **Fix:** correct the four facts; mechanize a *factual* guard (e.g. grep for closed review-IDs described as open) — **not** a completeness check, which would contradict the curation policy.
- **MED [security, CSP-mitigated] — the R3 `sanitize_source_url` fix is incomplete: an embedded tab bypasses the scheme check.** `php/includes/source_url.php:32` rejects `\r\n\0` but not TAB/C0; `j⇥avascript:` / `da⇥ta:text/html,…` make `parse_url` see no scheme (`:38`), so they fall through the relative-path branch (`:41`) unchanged, render as an `href` in the maintainer review UI (`review_handler.php:269`), and browsers strip the tab → live `javascript:`/`data:`. CSP `script-src 'self'` blocks `javascript:` execution and browsers block top-level `data:` nav, so defense-in-depth — but **the R3 fix did not fully stick** and `SourceUrlTest.php` still has no tab/control case. **Fix:** add tab to the existing filter → `/[\r\n\t\0]/` (browsers strip exactly tab/LF/CR per the WHATWG URL spec, so that closes the whole "browser-collapses-this" class, not just tab); add the regression test.
- **MED [security, defense-in-depth] — review-apply interpolates a column name with no allowlist re-check.** `php/includes/review_logic.php:123` builds `"$f = ?"` where `$f` is a key from the stored `payload_json` (`review_handler.php:160`). Safe today — `propose_handler.php` allowlists the keys at write time — but the *apply* path never re-validates, so any future writer that lands a non-allowlisted key turns a privileged `UPDATE reach` into SQL injection. **Fix:** intersect `$f` against an explicit reach-column allowlist before composing the SET clause.
- **LOW [correctness] — the incremental gauge-cache writer lacks the source-tiebreak the bulk rebuild has** *(downgraded from MED on prod cross-verification)*. `src/kayak/db/cache.py:178` orders `update_latest_gauge` by `observed_at DESC` only; `_BULK_REBUILD_GAUGE_CACHE_SQL:252` uses `observed_at DESC, source_id DESC`. But the bulk rebuild (pipeline step 4 `update-gauge-cache`) wipes and rebuilds the whole table and is the **last** cache writer before `build`, so a divergent incremental pick is overwritten within the same run — observable only between a bare `levels fetch` and the next full pipeline, and only for two sources at a byte-identical `observed_at`. Real wart, worth the 1-line fix, not user-facing. **Fix:** add `Observation.source_id.desc()` as the secondary sort (`:178`, `:199`).
